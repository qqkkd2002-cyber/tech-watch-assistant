# Tech Watch Tracker 결정 로그

이 문서는 Tech Watch Tracker가 지금 구조가 되기까지의 핵심 판단을 남긴다.

목적은 단순한 작업 기록이 아니다. 새 대화, 새 작업자, 또는 나중의 내가 이 문서만 보고도 “왜 이렇게 만들었는지”를 이어받을 수 있게 하는 것이다.

## 한 줄 정체성

Tech Watch Tracker는 필요할 때마다 리서치하는 도구가 아니라, 매일 신호를 미리 쌓아두고 AI가 1차 편집 초안을 올리는 사전 축적 도구다.

사용자는 모든 기사를 읽는 사람이 아니라 편집장이다.

```text
수집
→ 원문 읽기
→ 요약
→ AI 1차 후보 분류
→ 사용자가 확정/교정
→ 다음 판단과 보고서에 반영
```

## 처음 문제의식

초기 앱은 RSS, 문서, 키워드 뉴스를 모으고 보고서를 만드는 쪽에 가까웠다. 하지만 실제 사용 흐름에서는 세 가지 문제가 있었다.

1. 너무 많이 쌓이면 사용자가 다시 직접 읽어야 했다.
2. 키워드 매칭만으로는 “업무에 쓸 만한 신호”와 “그냥 걸린 뉴스”가 섞였다.
3. Gemini API 쿼터와 비용 문제 때문에 자동 요약/분류를 마음껏 돌리기 어려웠다.

그래서 방향을 바꿨다.

- 수집량을 늘리는 것보다, 쌓인 것 중 재사용 가치 있는 것을 가려내는 쪽으로 간다.
- AI가 최종 결정을 내리는 게 아니라 1차 편집 초안을 만든다.
- 사용자의 판정은 별도로 보존해서 AI 판정과 섞지 않는다.

## 큰 전환 1: Gemini 중심에서 Ollama/Gemma 중심으로

처음에는 Gemini API를 요약과 분류에 쓰는 흐름이었다. 하지만 운영 관점에서 문제가 있었다.

- 429 쿼터 제한이 자주 걸릴 수 있다.
- 새 기사가 쌓일 때마다 외부 API 비용과 제한을 신경 써야 한다.
- 사용자가 개인적으로 쌓는 전략 자료가 외부 API로 계속 나가는 구조가 부담스럽다.

그래서 로컬 Ollama + `gemma4:latest`를 핵심 경로로 바꿨다.

결정:

- 정밀 분류와 새 기사 요약의 기본 경로는 로컬 Ollama/Gemma를 우선한다.
- Gemini는 백업 또는 수동 경로 성격으로 남긴다.
- 로컬 모델이 실패하거나 근거가 부족하면 억지로 결론내지 않고 보류한다.

이 결정으로 얻은 것:

- 쿼터/비용 부담이 크게 줄었다.
- 민감할 수 있는 업무 맥락을 로컬에서 처리할 수 있다.
- 자동화 실험을 훨씬 자유롭게 할 수 있게 됐다.

## 큰 전환 2: 별표가 아니라 편집장 구조

초기에는 별표/저장 항목이 중요한 자료를 표시하는 중심이었다.

하지만 별표는 사용자가 이미 판단한 뒤의 결과다. AI가 먼저 후보를 올리고 사용자가 교정하는 구조에는 부족했다.

그래서 두 테이블을 분리했다.

```text
ai_editor_reviews
- AI의 1차 판단
- 업무 신호 / 학습 신호 / 노이즈 / 검토 대기
- 점수, 확신도, 이유, 태그, 모델명

editor_judgments
- 사용자의 최종 판정
- AI 판단을 승인/수정/반려한 기록
- 향후 학습 데이터
```

핵심 원칙:

- AI 판단은 초안이다.
- 사용자 판정은 확정이다.
- AI가 `manual_saved`나 사용자 판정을 덮어쓰지 않는다.
- 자동 분류를 켜더라도 `editor_judgments`는 사용자가 누를 때만 저장한다.

이게 나중에 Gemma 자동 분류를 붙일 때 가장 중요한 안전장치가 된다.

## 분류 기준

분류는 단순 키워드 매칭이 아니다.

예를 들어 “AI”, “금융”, “교육”, “협업” 같은 단어가 들어갔다고 바로 중요한 것은 아니다.

현재 기준:

### 업무 신호

사용자의 제품 전략, 경쟁사 분석, 제안/RFP, 고객 대응, 금융 IT 현대화, 규제 대응에 직접 재사용할 수 있는 항목.

예:

- 금융권 망분리 규제 완화와 AI/SaaS 도입 영향
- 은행권 AI 내부통제 체계
- 카카오뱅크/토스/금융권 IT 운영 사례
- 금융 클라우드, SaaS, 차세대, 보안 규제 변화

### 학습 신호

당장 업무에 직접 연결되지는 않지만, 나중의 전략 판단에 도움이 되는 기술 개념, 아키텍처, 운영 방식, 거버넌스 원칙.

예:

- RAG 아키텍처
- AI 에이전트 구현 패턴
- 플랫폼 엔지니어링 사례
- DevOps/SRE 운영 모델

### 노이즈

키워드는 걸렸지만 나중에 전략 자료로 재사용할 가능성이 낮은 항목.

예:

- 일반 행사/교육/채용/시상식
- 기술·제품·규제·시장 변화 없이 이름만 언급된 기사
- 연예/골프/공연/여행 등 검색 노이즈
- 제목과 설명에 검색 키워드의 핵심 단어가 전혀 없는 기사

### 검토 대기

요약이나 근거가 부족해서 AI가 확정하면 안 되는 항목.

원칙:

- 근거가 없으면 지어내지 않는다.
- 본문이 부족하면 본문 확보 대기로 보낸다.
- 판단하기 애매하면 낮은 확신도로 검토 대기에 둔다.

## 큰 전환 3: Google News는 제목만 보지 말고 원문을 읽는다

초기 트렌드 파이프라인은 Google News RSS의 제목/설명 중심이었다. 하지만 이 방식은 한계가 컸다.

- RSS 설명은 짧다.
- 원문 맥락이 없어 요약과 분류가 얕아진다.
- 관련기사나 검색 결과 노이즈가 섞일 수 있다.

그래서 새 기사에 대해 다음 흐름을 만들었다.

```text
Google News URL
→ 원문 URL 해석
→ 원문 본문 추출
→ Gemma 요약
→ 근거 저장
→ 후보 정리
```

중요한 안전장치:

- 본문 부족/추출 실패 시 `본문 확보 대기`
- 억지 요약 금지
- 원문 URL 기준 중복 제거
- 관련기사 오염 방지
- 요약 evidence 저장

실제 검증 결과:

- 실제 새 기사 큐 41건 중 중복 13건 제거
- 28건 처리
- 27건 성공, 1건 정직 보류
- 성공률 96.4%
- 원문 해석 실패 0%
- 요약 품질 양호

이 검증으로 “5건 시연이 아니라 실제 분량에서도 동작한다”는 것을 확인했다.

## 상한 처리 버그와 수정

자동화를 바로 켜지 않고 실제 분량 1회 점검을 한 이유가 여기서 드러났다.

처음 발견한 위험:

```text
한 주기 처리 상한을 넘은 기사
→ metadata/queued로 저장
→ 다음 주기에 이미 저장된 중복으로 오인
→ 영원히 요약되지 않음
```

이건 조용히 데이터가 새는 버그였다. 에러가 나지 않고, 사용자는 나중에 누락을 발견하기 어렵다.

수정 방향:

- `queued`와 `summarized`를 명확히 구분한다.
- `queued` 항목은 다음 주기 시작 시 먼저 처리한다.
- 원문 URL 중복 제거는 처리 완료 항목에는 적용하되, 대기 항목을 영구 누락시키지 않는다.
- 과거 `not_attempted` 백로그는 자동으로 건드리지 않는다.

검증:

- 상한 2건 시나리오에서 남은 1건이 다음 주기에 처리됨
- 운영 주기에서도 기존 queued 6건이 전부 요약 완료됨
- 새로 상한을 넘긴 항목은 다시 queued로 보존됨

이 수정 후 자동 ON 게이트를 열었다.

## 자동 ON 결정

자동 ON은 충동적으로 켠 것이 아니라 단계별 게이트를 통과한 뒤 켰다.

통과 조건:

- 실제 분량 성공률 96.4%
- 원문 해석 실패 0%
- 본문 부족은 정직 보류
- 상한 누락 버그 수정 및 재검증
- 연예/공연/골프 등 명백 노이즈 사전 차단
- 사용자 판정/`manual_saved` 보존
- 커밋/푸시 완료

현재 자동 요약 범위:

```text
자동 수집
→ 원문 해석
→ 본문 추출
→ Gemma 요약
→ evidence 저장
→ queued 자동 처리
```

아직 자동 분류까지 완전 연결한 것은 아니다.

## 새로 추가한 검색 노이즈 방어

운영 관찰 중 `AI 거버넌스` 키워드에 전혀 맞지 않는 연예성 기사가 요약까지 가는 사례가 발견됐다.

원인:

- 기존 노이즈 필터는 제목 금칙어 중심이었다.
- “연예” 같은 단어가 제목에 없으면, 검색 키워드와 무관해도 통과할 수 있었다.

수정:

- 검색 키워드의 핵심 토큰을 뽑는다.
- 제목/설명에 핵심 토큰이 하나도 없으면 요약 전에 `noise_filtered`로 내린다.

예:

```text
keyword: AI 거버넌스
title: 노주현, 서울 마곡 초호화 실버타운...
→ keyword_mismatch 로 noise_filtered
```

주의:

- 이 필터는 보수적으로 적용한다.
- 제목/설명에 핵심 토큰이 하나라도 있으면 요약 기회를 준다.
- 너무 공격적으로 필터링해서 진짜 신호를 버리지 않도록 한다.

## UI 사용성 결정: 요약은 끝까지 볼 수 있어야 한다

후보 카드에서 AI 요약이 4줄로 잘려 보이고 `더보기`가 없는 문제가 있었다.

이건 작은 UI 문제가 아니라, 자동 요약의 가치를 사용자가 확인하지 못하는 문제였다.

수정:

- 후보 카드의 긴 AI 요약에 `더보기` 버튼 추가
- 누르면 전문 펼침
- 다시 누르면 접기
- 짧은 요약에는 버튼을 표시하지 않음

## 지금 켜진 것과 아직 안 켜진 것

### 켜진 것

- 자동 수집
- 새 트렌드 원문 해석
- 본문 추출
- Gemma 요약
- 회당 처리 상한
- queued 보존 및 다음 주기 우선 처리
- 실패율 30% 경고
- 명백 노이즈 사전 차단
- 키워드 불일치 노이즈 사전 차단
- 사용자 판정/`manual_saved` 보존

### 아직 안 켜진 것

- Gemma 자동 정밀 분류 완전 연결
- 과거 364건 트렌드 백로그 소급
- 과거 문서/후보 대량 소급
- 보고서가 자동 분류 확정 신호를 강하게 반영하는 단계

## 과거에 쌓인 데이터 처리 원칙

과거 대량 백로그는 아직 자동 처리하지 않는다.

이유:

- 새 자동 흐름이 먼저 안정돼야 한다.
- 과거 데이터에는 오래된 링크, 깨진 Google News URL, 낮은 품질의 검색 결과가 섞여 있을 수 있다.
- 한 번에 소급하면 품질 확인과 노이즈 관리가 어렵다.

현재 원칙:

- 새로 들어오는 기사와 새 queued만 자동 처리한다.
- 과거 `not_attempted` 백로그는 OFF로 유지한다.
- 안정화 후 별도 작업으로 소량씩 소급한다.
- 사용자 판정과 `manual_saved`는 절대 덮어쓰지 않는다.

## 앞으로 남은 작업

### 1. 3~7일 자동 요약 관찰

확인할 것:

- queued가 다음 주기에서 잘 빠지는지
- 실패율이 30% 근처로 올라가지 않는지
- 요약 품질이 유지되는지
- 검색 노이즈가 줄어드는지
- 사용자 판정/저장 항목이 보존되는지

### 2. Gemma 자동 분류 연결

자동 분류는 단순 스위치처럼 켜면 안 된다.

원칙:

- Gemma는 초안을 배치한다.
- 사용자가 확정한다.
- 자동 분류 결과와 사용자 확정 결과를 UI에서 구분한다.
- `ai_editor_reviews`에는 자동 분류를 저장한다.
- `editor_judgments`는 사용자가 누를 때만 저장한다.
- 근거 부족/요약 부족 항목은 `review_queue`에 둔다.

필요한 UI:

- “Gemma 자동 분류 · 미확인”
- “사용자 확정”
- 자동 분류 결과를 검수할 수 있는 카드 표시

### 3. 과거 백로그 소량 소급

자동 요약과 자동 분류가 안정된 뒤 진행한다.

순서:

1. 최근/품질 좋은 과거 항목 일부만 샘플 소급
2. 품질 확인
3. 문제 없으면 묶음 단위로 확장
4. 오래된 Google News 링크는 최후순위

## 운영 메모

- 이 앱은 로컬 맥북에서 돈다.
- 자동화가 돌려면 맥북, 서버, Ollama가 켜져 있어야 한다.
- 자동 수집 시각은 마지막 수집 시각 + 24시간 기준이다.
- 수동으로 당겨 실행하면 다음 자동 실행 시각도 그만큼 밀린다.
- 저녁 7시처럼 고정 시각 운영이 필요하면 별도 스케줄 기준을 구현해야 한다.

## 커밋 기준

검증된 변경은 로컬에 오래 두지 않는다.

이번 프로젝트에서 확인한 원칙:

- 실제 분량에서 통과한 지점은 커밋한다.
- 안전장치 수정과 재검증이 끝나면 커밋한다.
- 자동 ON 같은 운영 분기점은 반드시 커밋/푸시한다.

중요 커밋:

```text
6e53b80 트렌드 본문 읽고 요약 파이프라인 완성
25fbe5b 트렌드 본문 요약 자동 파이프라인 활성화
7185a05 UTC→서울 시간대 보정
b6f1fd1 검색 키워드 불일치 노이즈 차단
ec95e9b 후보 카드 AI 요약 더보기 추가
```

## 이 프로젝트에서 지킨 방식

이 프로젝트는 한 번에 완전 자동으로 간 것이 아니라, 매번 한 칸 먼저 재보고 다음으로 갔다.

반복한 방식:

```text
아이디어
→ 작은 시연
→ 실제 분량 검증
→ 결함 발견
→ 코드로 안전장치화
→ 재검증
→ 커밋
→ 다음 단계
```

이 방식 덕분에 다음 문제들을 자동 ON 전에 잡았다.

- Gemini 쿼터 의존
- 키워드 기반 분류의 낮은 정확도
- 본문 없는 트렌드 요약의 한계
- 상한 초과 기사의 영구 누락
- 연예/검색 노이즈 요약
- UTC/KST 시간대 혼선
- 카드에서 요약 전문을 못 읽는 사용성 문제

## 마지막 원칙

자동화의 목적은 사용자를 배제하는 것이 아니다.

사용자가 매일 모든 기사를 읽지 않아도 되게 하는 것이다.

AI는 초안을 만들고, 사용자는 편집장으로 판단한다. 이 원칙은 자동 요약과 자동 분류가 더 강해져도 유지한다.
