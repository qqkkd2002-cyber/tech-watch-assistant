# Tech Watch Tracker v2

솔루션전략팀이 나중에 전략 보고서, 상위 기획, 제안서 또는 제품 전략을 작성할 때 바로 꺼내 쓸 수 있도록 기술·경쟁 정보를 미리 수집하고 정제하는 로컬 웹앱입니다.

> 필요한 순간에 리서치를 시작하는 대신, 매일 자동으로 전략 자료를 모으고 정제해 미리 쌓아둡니다.

전략 보고서는 이 앱의 핵심이 아니라, 축적된 자료로 만들 수 있는 결과물 중 하나입니다.

## 만든 이유

기존에는 전략 문서가 필요해진 뒤에야 관련 기사와 경쟁사 자료를 다시 검색하고 정리해야 했습니다. 이 과정은 시간이 많이 들고, 조사한 자료도 개인에게 흩어지기 쉬웠습니다.

Tech Watch Tracker는 이 순서를 뒤집습니다.

```text
자동 수집 -> AI 후보 정리 -> 사용자 판단 -> 전략 자료 축적 -> 기획·제안·보고서에 재활용
```

## 핵심 운영 방식

### 1. 자동 수집

프로필에 등록한 RSS, 경쟁사 제품 피드, 기술 블로그, 레퍼런스 피드와 관심 키워드를 기준으로 자료를 수집합니다. 발행일과 수집일을 따로 기록하며, 자동 수집과 사용자가 실행하는 `지금 수동 수집`은 별개로 동작합니다.

### 2. 후보 정리

수집된 모든 자료를 사용자가 직접 읽는 대신, 먼저 다시 활용할 가능성이 있는 후보로 정리합니다. 판단 기준은 단순한 키워드 일치가 아니라 다음 질문입니다.

> 이 자료를 향후 전략 보고서, 상위 기획, 제안 또는 제품 전략에 다시 꺼내 쓸 가치가 있는가?

현재 기본 후보 생성은 규칙 기반이며, LLM 정밀 분류를 제한된 범위에서 검증하는 단계입니다.

### 3. 편집 판단

`후보 정리`는 새로 들어온 자료를 판단하는 작업대입니다. 사용자는 후보를 다음 상태로 나눕니다.

- **업무 신호**: 담당 제품, 경쟁사, 시장 전략이나 제안 업무에 직접 활용할 자료
- **학습 신호**: 당장 담당 업무와 직접 연결되지는 않지만 기술·시장 이해에 도움이 되는 자료
- **노이즈**: 현재 업무와 학습에 활용할 가치가 낮은 자료
- **검토 대기**: 아직 사용자가 판단하지 않은 자료

노이즈로 판단해도 원본은 삭제되지 않습니다. 모든 수집 자료는 `수집 원장`에서 다시 검색하고 확인할 수 있습니다.

### 4. 축적과 재활용

업무 신호와 학습 신호로 확정한 자료는 보관 공간에 연결됩니다. 사용자가 직접 누른 별표와 편집 판단은 독립적으로 관리되므로, 한 자료가 수동 저장과 업무 신호 상태를 동시에 가질 수 있습니다.

축적된 자료는 경쟁사·기술 동향 확인과 전략 보고서 생성에 다시 활용합니다.

## 주요 화면

- **후보 정리**: 아직 판단하지 않은 새 후보를 검토하는 작업대
- **대시보드**: 경쟁사 활동, 기술 테마와 수집 현황을 요약하는 화면
- **수집 원장**: 자동·수동 수집으로 들어온 전체 원본 자료
- **중요 보관함**: 수동 저장, 업무 신호와 학습 신호를 저장 이유별로 확인하는 공간
- **전략 보고서**: 축적된 자료를 활용해 보고서를 생성하고 보관하는 부가 기능
- **프로필 설정**: 관심 키워드·폴더, RSS, 경쟁사/레퍼런스 피드와 자동 수집 설정 관리

## 개인화와 AI 맥락

프로필별로 관심 키워드와 폴더, 수집 피드, 자동 수집 주기를 분리합니다. LLM 정밀 분류에서는 별도의 자유서술형 업무 소개가 아니라, 프로필에 실제 등록된 키워드·폴더와 경쟁사/레퍼런스 피드를 조합해 사용자의 업무 맥락으로 활용합니다.

## 현재 개발 상태

### 운영 중

- 자동·수동 자료 수집
- 프로필별 자동 수집 ON/OFF와 수집 주기
- 발행일과 수집일 분리 기록
- 경쟁사 문서, 기술 레퍼런스와 키워드 뉴스 수집 원장
- 개별 AI 요약 생성과 요약 대기 재시도
- 후보 정리 보드와 드래그 이동
- 업무 신호·학습 신호·노이즈 판단 및 사용자 판단 이력 저장
- 수동 별표(`manual_saved`)와 편집 판단(`editor_bucket`) 상태 분리
- 확정 신호와 중요 보관함 연동
- 후보 정리와 대시보드 화면 분리
- 전략 보고서 생성·아카이빙
- Discord 알림 연동

### 구현 및 소규모 검증 완료

- 프로필의 키워드·폴더·피드를 반영한 LLM 기사 단위 정밀 분류
- 정해진 태그 목록 안에서만 태그를 제안하는 분류 구조
- 기사별 분류 이유, 전략 가치 점수와 확신도 저장
- 사용자 판단이 있는 자료를 LLM이 덮어쓰지 않는 보호 로직
- LLM이 노이즈로 분류한 항목을 다시 확인하고 신호로 올릴 수 있는 감시 목록 UI
- 로컬 Ollama `gemma4:latest` 기반 정밀 분류와 근거 부족 항목 차단
- Google News 중계 URL을 실제 원문 URL로 해석하는 교체형 URL 해석기
- 원문 본문 추출, Gemma 한국어 요약, 근거 문장 검증 파이프라인
- 원문 URL 기준 중복 차단과 여러 검색 키워드 병합
- 본문 확보 실패율 기록, 시스템 로그와 사이드바 경고

새 트렌드 파이프라인은 운영 DB를 변경하지 않는 실제 최신 큐 28건 점검에서 27건이 원문 해석부터 요약까지 완료됐고, 본문을 신뢰할 수 없었던 1건은 `본문 확보 대기`로 분기됐습니다. 원문 URL 해석 실패는 0건, 전체 실패율은 3.6%였습니다.

수집 주기 상한을 넘긴 새 기사는 `queued` 상태로 보존하며, 다음 주기 시작 시 새 RSS 결과보다 먼저 처리합니다. 따라서 상한 초과 항목이 중복으로 버려져 영구 누락되지 않습니다. 명백한 연예·공연·골프 제목은 원문 추출과 Gemma 요약 전에 보수적으로 차단합니다. 과거 `not_attempted` 자료는 이 큐에 포함하지 않으므로 기존 트렌드 소급은 계속 꺼져 있습니다.

새 기사 본문·요약 파이프라인은 기본 ON이며 수집 주기당 최대 10건을 처리합니다. 환경 변수 `TWT_TREND_PIPELINE_ENABLED=0`으로 비상 중지할 수 있고, `TWT_TREND_PIPELINE_LIMIT`으로 주기당 상한을 조정할 수 있습니다. 업무/학습 신호의 자동 정밀 분류와 과거 자료 소급은 별도 단계로 유지합니다.

### 다음 검증

- 새 트렌드 본문·요약 파이프라인의 제한된 운영 적용
- 새 요약을 이용한 수집 시 정밀 분류 범위 결정
- 본문이 충분한 기존 문서의 요약·분류 소급
- 과거 수집 자료 소급 분류의 일일 처리량 결정

따라서 현재는 `수집 -> 원문 확보 -> 로컬 AI 요약 -> 근거 기반 정밀 분류` 구조를 소량 검증한 상태이며, 전체 자동 운영을 켜기 전 제한 적용 단계입니다.
