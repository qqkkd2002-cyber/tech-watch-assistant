# Tech Watch Tracker 작업 과정 기록

이 문서는 `Tech Watch Tracker` 프로젝트를 포트폴리오로 정리하거나, 다른 AI에게 프로젝트 맥락을 전달하기 위해 작성한 작업 과정 요약입니다.

실제 대화 전문을 그대로 옮기지는 않았습니다. API Key, Discord Webhook, 로컬 DB, 내부 설정값 같은 민감 정보가 포함될 수 있기 때문입니다. 대신 포트폴리오 작성에 필요한 목적, 문제의식, 의사결정, 구현 과정, 결과물을 안전하게 정리했습니다.

## 1. 프로젝트 시작 배경

사용자는 Antigravity를 활용해 로컬 웹 애플리케이션 형태의 `Tech Watch Tracker`를 먼저 구현해 두었습니다.

초기 화면은 다음과 같은 기능을 포함하고 있었습니다.

- 경쟁사 RSS 피드 설정
- 기술 키워드 설정
- Gemini API Key 입력
- Discord Webhook URL 입력
- 모니터링 피드
- 중요 보관함
- 월간 전략 보고서
- 프로필 설정
- 사용 설명서
- 시스템 로그

하지만 Antigravity 환경이 먹통이 되면서, 구현한 내용이 사라지거나 더 이상 관리하기 어려워질 수 있다는 불안이 있었습니다. 사용자는 비개발자였기 때문에 로컬 개발, GitHub, 서버, 형상관리 개념을 처음부터 이해해야 했습니다.

## 2. 처음 정리한 개발 방향

처음에는 다음 질문을 중심으로 방향을 정리했습니다.

- 로컬에서 작업하는 것이 맞는가?
- GitHub에 올리면 앱이 운영되는 것인가?
- GitHub와 로컬 서버는 무엇이 다른가?
- 팀원들과 나중에 형상관리할 수 있는가?
- Codex 또는 Antigravity가 중단되어도 앱이 계속 살아 있으려면 어떻게 해야 하는가?

정리한 결론은 다음과 같습니다.

- 앱이 실제로 실행되는 곳은 로컬 맥북 또는 별도 서버입니다.
- GitHub는 앱을 실행하는 곳이 아니라 소스코드를 백업하고 이력을 관리하는 공간입니다.
- 지금 단계에서는 로컬 파일을 기준으로 개발하고, GitHub에는 안전한 소스코드만 올리는 방식이 적합합니다.
- API Key, Discord Webhook, SQLite DB, 로그 파일은 GitHub에 올리면 안 됩니다.

## 3. 프로젝트의 핵심 목적

프로젝트 목적은 단순한 뉴스 수집기가 아니라, 솔루션전략팀의 반복 리서치 업무를 자동화하는 것입니다.

핵심 목적:

- 경쟁사 릴리즈와 기술 트렌드를 자동 수집
- AI로 핵심 요약 생성
- Discord로 실시간 공유
- 주간/월간 전략 보고서 생성
- 누적 데이터를 팀의 전략 리서치 자산으로 축적
- 경쟁사 비교자료, 제품 포지션 비교, 기술 채택 흐름 분석에 활용

중요한 운영 메시지는 다음과 같이 정리했습니다.

> 개인별 관심사는 프로필로 분리하고, 수집 데이터와 분석 산출물은 팀 자산으로 중앙 축적한다.

## 4. 로컬 프로젝트 정리

Antigravity에서 작업한 앱을 로컬 프로젝트로 이어갈 수 있도록 파일 기반 구조를 정리했습니다.

프로젝트 폴더:

```text
/Users/parkjongho/Documents/New project/tech-watch-assistant
```

주요 파일:

- `web_server.py`: FastAPI 기반 웹 서버
- `agent.py`: 수집, AI 요약, 보고서 생성 담당
- `database.py`: SQLite DB 스키마와 조회/저장 로직
- `monitors.py`: RSS/Google News 수집
- `discord_notifier.py`: Discord Webhook 알림
- `static/index.html`: 웹 UI
- `static/app.js`: 프론트엔드 동작
- `static/style.css`: 스타일
- `README.md`: 프로젝트 설명
- `config.example.json`: 설정 예시

## 9. 보고서 생성 방식 개선

보고서 생성은 수동/자동을 분리해야 한다는 의견이 나왔습니다.

반영한 방향:

- 데이터 수집은 백그라운드에서 자동 실행 가능
- 보고서 생성은 자동 또는 수동 중 선택 가능
- 프로필 설정에서 자동 보고서 생성 여부를 선택
- 수동 보고서는 사용자가 버튼을 눌러 즉시 생성

또한 보고서를 전체 데이터 기준으로만 작성하는 것은 부족하다고 판단했습니다.

추가한 범위 선택:

- 특정 폴더 기준 보고서
- 여러 폴더 선택
- 특정 키워드 기준 보고서
- 여러 키워드 선택
- 키워드 조건: 하나라도 포함 또는 모두 포함

이를 통해 `전체 누적 데이터`뿐 아니라 특정 관심 영역만 대상으로 전략 보고서를 만들 수 있게 했습니다.

이로써 앱의 가치를 `보고서 생성 도구`에서 `팀 리서치 자산 축적 도구`로 확장했습니다.

## 11. 피드 수집의 최신성 문제 해결

스캔을 누르면 최신 기사가 아니라 오래된 기사도 가져오는 문제가 발견되었습니다.

문제:

- 수집일과 발행일이 구분되지 않음
- 오래된 기사가 오늘 수집되면 최신 기사처럼 보임
- Discord 알림도 오래된 기사에 대해 발송될 수 있음

해결 방향:

- 발행일 `published_at` 저장
- 수집일 `created_at`과 발행일 분리
- 피드 정렬은 발행일 우선
- Discord 알림은 발행일 기준 최신 항목만 발송
- Google News 검색에 최근성 조건 추가
- 화면 카드에 발행일과 수집일을 분리 표시

추가로 기존 RSS 날짜 형식이 다음처럼 영문 문자열로 들어오는 문제가 있었습니다.

```text
Mon, 01 Jun 2026 14:00:26 +0000
```

이를 화면에서 날짜만 깔끔하게 표시하도록 변환했습니다.

최종 표시 형태:

```text
발행일 2026. 06. 01.
수집일 2026. 06. 02.
```

시간과 분은 전략 업무에서 크게 필요하지 않다고 보고 제거했습니다.

## 14. GitHub 형상관리 과정

사용자는 GitHub 계정을 만들고 웹에서 로그인했습니다.

처음에는 GitHub Desktop을 설치하고 연결하는 과정을 안내했습니다.

정리한 개념:

- GitHub 웹사이트: 저장소 확인, 설정, 공유
- GitHub Desktop: 로컬 폴더와 GitHub 저장소를 연결하고 커밋/푸시하는 도구
- 로컬 웹앱: 실제 실행되는 앱 본체

처음에는 실수로 상위 폴더인 `New project`가 Git 저장소로 잡혔습니다.

문제:

- `tech-watch-assistant`뿐 아니라 상위 폴더의 다른 파일도 같이 올라갈 수 있었음
- 포트폴리오와 무관한 파일이 섞일 수 있었음

조치:

- 상위 폴더의 잘못된 `.git` 연결을 백업 이름으로 변경
- `tech-watch-assistant` 폴더 안에서 새 Git 저장소 초기화
- `.gitignore`를 프로젝트 폴더 안에 추가
- GitHub Desktop에서 `tech-watch-assistant` 폴더를 다시 선택

최종 GitHub 저장소:

```text
tech-watch-assistant
```

초기 커밋:

```text
Initial commit for Tech Watch Tracker
```

저장소는 Private으로 생성했습니다.

## 17. 포트폴리오에서 강조할 의사결정

포트폴리오 작성 시 단순히 `AI 앱을 만들었다`가 아니라, 다음 포인트를 강조하면 좋습니다.

### 문제 정의

솔루션전략팀은 경쟁사 릴리즈와 기술 트렌드를 계속 추적해야 하지만, 수작업 검색과 요약은 시간이 많이 들고 누락 가능성이 큽니다.

### 해결 방향

RSS, 키워드 뉴스, AI 요약, Discord 알림, 전략 보고서 생성을 하나의 흐름으로 묶어 반복 리서치 업무를 자동화했습니다.

### 업무 관점 설계

단순 뉴스 수집이 아니라, 반복 신호, 키워드 빈도, 경쟁사 방향성, 제품 포지션 비교, 다음 액션 제안까지 이어지는 전략 업무 흐름을 설계했습니다.

### 사용자/팀 운영 구조

팀원은 각자 프로필로 관심사를 운영하고, 수집 데이터와 분석 결과는 팀 자산으로 축적되는 구조를 고려했습니다.

### 비개발자 관점의 구현 경험

비개발자로서 로컬 개발, 서버 실행, API 연동, GitHub 형상관리까지 하나씩 익히며 실제 작동하는 업무 도구를 만들었습니다.
